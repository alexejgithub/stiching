Type: research
Status: resolved

## 1. Algorithm options

There are five algorithm families that show up repeatedly in both the academic literature and the JS libraries surveyed in section 2. A 2023 survey, M. Emre Celebi, "Forty years of color quantization: a modern, algorithmic survey," *Artificial Intelligence Review* (2023) — DOI page at [Springer](https://link.springer.com/article/10.1007/s10462-023-10406-6) / [ACM DL mirror](https://dl.acm.org/doi/abs/10.1007/s10462-023-10406-6) — traces the field back to Heckbert's 1980/1982 median-cut work and covers median-cut, octree, k-means/minmax, and fuzzy c-means as the major families. **Caveat:** both the Springer and ACM pages returned an authentication redirect / 403 when fetched directly, so I could not read the full text — I'm citing it only for its existence, scope, and the historical framing ("landmark median-cut algorithm... published by Heckbert in 1980"), which was corroborable via search-result abstract snippets, not for any detailed claim inside it.

### Popularity / histogram algorithm
Build a histogram of the image's colors (usually after truncating to fewer bits per channel to keep the histogram small) and keep the N most frequent buckets, one output color per bucket. Described in the University of Edinburgh's HIPR2 image-processing glossary: "creates a histogram of all colors and retains the 256 most frequent ones" ([homepages.inf.ed.ac.uk/rbf/HIPR2/quantize.htm](https://homepages.inf.ed.ac.uk/rbf/HIPR2/quantize.htm)), which also notes it predates and is simpler but lower-quality than median-cut ("median-cut algorithm yields even better results but also needs more computation time").
- **Speed:** single O(N) pass over pixels, no sorting or iteration — the cheapest option by construction.
- **Quality at k=1-8 (my own reasoning, not sourced from the above page, which only discusses k=256):** this is the riskiest algorithm at very small k. Because it only counts raw frequency, a photo dominated by, e.g., many close shades of skin tone or sky can fill all 8 buckets with near-duplicate colors while a small-but-visually-important patch (a red scarf, a logo) is simply outvoted and disappears entirely. Median-cut/octree/k-means all instead split or cluster by *spread* in color space, which tends to preserve visually distinct minority colors better. Popularity is a reasonable fast fallback but is not what I'd pick as the primary algorithm for a feature whose whole point is picking a good small palette.

### Median cut
Paul Heckbert, "Color Image Quantization for Frame Buffer Display," *Proc. SIGGRAPH '82* (1982), pp. 297-307 — landing page at the [SIGGRAPH History project](https://history.siggraph.org/learning/color-image-quantization-for-frame-buffer-display-by-heckbert/), paper indexed at [ResearchGate](https://www.researchgate.net/publication/2310359_Color_Image_Quantization_for_Frame_Buffer_Display). Recursively finds the bounding box of all colors in use, splits it at the median along its longest axis so each half contains roughly equal pixel counts, and repeats until the target color count is reached; each box's centroid becomes an output color. A widely mirrored technical write-up (originally Dr. Dobb's, Sept 1994) walks through the implementation and states the dominant cost is the per-box sort needed to find the median: "quicksort... O(N log₂N)" average case, degrading to O(N²) worst case on adversarial (pre-sorted) input, with heapsort suggested as a safer O(N log N) worst-case alternative — [jacobfilipp.com mirror of the Dr. Dobb's article](https://jacobfilipp.com/DrDobbs/articles/DDJ/1994/9409/9409e/9409e.htm).
- **Speed:** for a target of k colors you only need `ceil(log2(k))` splits (k=8 -> 3 splits, k=1 -> 0 splits, i.e. just the average color). Because splitting only requires ~log2(k) rounds, total cost is dominated by the pixel/histogram sort at the start, not by k — meaning **going from k=8 to k=1 barely changes runtime**, all the cost is in ingesting the pixels once.
- **Quality:** the standard "good default." A refinement called Modified Median Cut Quantization (MMCQ) is what Leptonica, and in turn the `quantize` and `colorthief` npm packages (see §2), actually implement — I attempted to fetch Leptonica's own description page (`leptonica.org/color-quantization.html`) but it failed with a TLS handshake error from this environment (`SSL routines: WRONG_VERSION_NUMBER`), so that specific claim about *why* Leptonica modified vanilla median-cut is not independently confirmed here; what is directly confirmed is the `quantize` npm package's own source code (fetched directly, see §2), which implements population- and volume-weighted box splitting via a priority queue plus an explicit fallback path for when very few colors are requested — behavior consistent with "modified to behave better at small k," directly relevant to a k=1-8 use case.

### Octree quantization
Michael Gervautz & Werner Purgathofer, "A Simple Method for Color Quantization: Octree Quantization," in *New Trends in Computer Graphics* (Magnenat-Thalmann & Thalmann, eds.), Springer, 1988 — [Springer chapter page](https://link.springer.com/chapter/10.1007/978-3-642-83492-9_20), also indexed in the Graphics Gems ACM DL entry [dl.acm.org/doi/10.5555/90767.90833](https://dl.acm.org/doi/10.5555/90767.90833). Inserts every pixel's color into an octree keyed on RGB bits (depth 8), then reduces the tree bottom-up, merging sibling leaves into their parent whenever the leaf count exceeds the target palette size, so that "every pixel has a well defined maximum error."
- **Speed:** comparable to median-cut — one insertion pass plus a bounded reduction pass, no sorting required, generally considered at least as fast as median-cut and more memory-efficient (per the chapter's own framing, corroborated via search-result abstract).
- **Quality at k=1-8 (my own reasoning — the source paper's examples target larger palettes, ~16-256):** the reduction step is driven purely by "how many leaves are left," triggered whenever the leaf count exceeds k. At k=8 this merge pressure is extreme and happens very early in the tree (near the root), which risks merging visually distinct colors together mechanically rather than by a variance-aware choice the way median-cut's split-at-median or Wu's variance-minimizing bipartition do. I'd treat octree as roughly tied with median-cut on speed but slightly riskier on quality at these very small k values, and it is also somewhat more code to implement correctly (tree insertion + bottom-up reduction with a running "reducible nodes" list) than median-cut.

### Wu's variance-minimizing quantizer
Xiaolin Wu, "Color Quantization by Dynamic Programming and Principal Analysis," *ACM Transactions on Graphics* — abstract at [dl.acm.org/doi/abs/10.1145/146443.146475](https://dl.acm.org/doi/abs/10.1145/146443.146475); the commonly-implemented fast variant is documented in the reference C source `cq.c` (Graphics Gems II, pp. 126-133), nominally hosted at [ece.mcmaster.ca/~xwu/cq.c](https://www.ece.mcmaster.ca/~xwu/cq.c) (direct fetch failed here with a TLS certificate error) and successfully fetched from a mirror at [gist.github.com/bert/1192520](https://gist.github.com/bert/1192520). The header describes the method as "greedy orthogonal bipartition of RGB space for variance minimization aided by inclusion-exclusion tricks," explicitly trading nearest-neighbor accuracy for speed: "For speed no nearest neighbor search is done. Slightly better performance can be expected by more sophisticated but more expensive versions."
- **Speed:** the inclusion-exclusion/moment trick lets each candidate split's resulting variance be computed in O(1) from precomputed cumulative moments, so the whole quantization is a small number of O(1) evaluations over a fixed-size 3D histogram grid (not the raw pixel count) plus one O(N) histogram-building pass. This is the algorithm `image-q`'s `WuQuant` option implements (see §2).
- **Quality:** widely regarded as one of the best speed/quality tradeoffs for small-to-medium palettes; it is one of the algorithms `image-q`'s own README lists by name alongside NeuQuant and RGBQuant, and is reported in secondary sources (not independently re-verified here) as the default quantizer in tools like ImageMagick and libimagequant/pngquant.

### k-means (Lloyd's algorithm) in color space
Standard iterative clustering: assign each pixel to its nearest of k centroids, recompute centroids as the mean of assigned pixels, repeat to convergence. Brute-force per-iteration cost is O(N·k) (N pixels, k clusters); efficient kd-tree-accelerated variants are described in Kanungo, Mount, et al., "An Efficient k-Means Clustering Algorithm: Analysis and Implementation" — [cs.umd.edu/~mount/Projects/KMeans/pami02.pdf](https://www.cs.umd.edu/~mount/Projects/KMeans/pami02.pdf) (the fetch tool could not decode this PDF's text content in this environment, so I'm citing it only for the paper's title/existence, which is independently well known in the clustering literature, not for any specific number pulled from its body). Its applicability specifically to color quantization is discussed in a dedicated paper, "Improving the Performance of K-Means for Color Quantization," abstract at [arxiv.org/abs/1101.0395](https://arxiv.org/abs/1101.0395), which states plainly that k-means "has not received much respect in the color quantization literature because of its high computational requirements and sensitivity to initialization," but that "an efficient implementation of k-means with an appropriate initialization strategy can in fact serve as a very effective color quantizer."
- **Speed:** at full photo resolution (millions of pixels) and multiple iterations to convergence, this is the most expensive of the five families. But — and this matters a lot for this project — cost scales with N (pixel count) times iterations, not directly with k in any way that dominates at k≤8; and per §3, N should already be small (a downscaled sample buffer) by the time quantization runs, at which point O(N·k·iterations) is cheap even for k-means.
- **Quality:** potentially the best of the five, because it directly minimizes intra-cluster variance in whichever color space you choose to cluster in (e.g. CIE Lab for perceptual uniformity, rather than raw RGB), which matters more at very small k where each of the 8 colors has to "cover" a large, perceptually varied part of the image. Its two weaknesses (slow at scale, sensitive to bad initialization/local minima) are both mitigated by (a) running it only against a small downscaled sample and (b) seeding its initial centroids from a fast pass of median-cut or Wu's algorithm instead of random initialization — a common hybrid pattern.

### Summary table (my synthesis of the above, not a single source)

| Algorithm | Per-run cost driver | Cost sensitivity to k=1 vs k=8 | Quality risk at k≤8 | Implementation complexity |
|---|---|---|---|---|
| Popularity/histogram | O(N) single pass | none | High — frequency-only, drops rare-but-important hues | Very low |
| Median cut (MMCQ) | O(N log N) sort of unique colors | Low (only log2(k) splits) | Low-medium — standard safe default | Low-medium (~150-550 LOC, see §2) |
| Octree | O(N) insert + bounded merge | Low | Medium — aggressive early merging at tiny k | Medium |
| Wu (variance-minimizing) | O(N) histogram + O(1) per split via moments | Very low | Low — strong quality/speed balance | Medium-high (moment math) |
| k-means / Lloyd's | O(N·k) per iteration × iterations | Scales with k, but k≤8 keeps it cheap once N is small | Low (best-in-class if well-seeded) | Medium (plus needs a seeding strategy) |

**Bottom line for k=1-8 specifically:** median-cut/MMCQ is the standard, lowest-risk choice — its cost is already dominated by a single ingest pass regardless of k, it needs no iteration, and (per the `quantize` source, §2) mainstream implementations already special-case small k. Wu's algorithm is a credible step up in quality for similar speed if a library that implements it (`image-q`) turns out to be worth its integration cost. k-means is worth keeping in mind as a "best quality" upgrade path specifically because this project already needs to downscale for performance (§3), which is exactly the precondition that makes k-means' main weakness (cost) stop mattering.

Notes on individual packages, from their own docs/source:

- **`colorthief`** ([lokesh/color-thief](https://github.com/lokesh/color-thief)) is the most actively maintained and highest-adoption option that is purpose-built for exactly "give me the dominant colors of an image, small k." It was republished 13 days before this research (2026-07-01), has 13,599 GitHub stars and only 1 open issue, and its README states the `colorCount` option accepts **2-20** colors and defaults its `quality` (pixel-sampling) option to **10**, documented verbatim as "Sampling rate (1 = every pixel, 10 = every 10th)" — i.e. **the library's own default behavior is to not process every pixel**, which is directly relevant evidence for §3. It uses the MMCQ algorithm lineage under the hood (same family as `quantize`), and the current v3 also added an OKLCH color-space option for more perceptually even palettes. It accepts `ImageData`/`ImageBitmap`/`OffscreenCanvas`/`HTMLCanvasElement` natively, so no manual pixel-array extraction is needed. Runtime dependencies are zero required — `sharp` is listed only as an **optional** peer dependency used for Node-side decoding, never invoked in-browser. One gap: its documented `colorCount` floor is 2, not 1, so a k=1 "average color" mode needs to be handled as a one-line special case in app code regardless of library choice.
- **`quantize`** ([source: lokesh/quantize](https://github.com/lokesh/quantize) — note this is the same GitHub author/org as `colorthief`, and is the standalone extraction of the MMCQ algorithm that used to be embedded directly in `color-thief`) is the smallest, simplest, most single-purpose option of the five: zero dependencies, 1.9 KB min+gzip. Fetching its source directly (`src/quantize.js`) confirmed it implements Modified Median Cut Quantization with `VBox`/`PQueue`/`CMap` data structures, ~550 lines total including a `SimpleColorMap` fallback path specifically for when very few colors are requested — i.e. **this library was already built with small-k robustness in mind**. Its GitHub repo was last pushed 2024-10-04 (not abandoned), but it has no TypeScript types and requires the caller to build the `[r,g,b][]` pixel array from `ImageData` manually.
- **`image-q`** ([GitHub](https://github.com/ibezkrovnyi/image-quantization)) bundles multiple algorithms behind one API — its own README lists "NeuQuant (...), NeuQuantFloat (...), RGBQuant, WuQuant" as selectable palette quantizers, i.e. **it includes an implementation of Wu's algorithm**, the family flagged in §1 as a strong quality upgrade path for small k. 167 GitHub stars, MIT-licensed per the README's own text (the GitHub API's machine-readable `license` field came back `null`, which looks like a metadata/detection gap rather than a real licensing question, since the README explicitly states "MIT License"). Its ~3.6M/week download number is far higher than any other package here — high enough that it's almost certainly a transitive dependency of some other popular tool rather than 3.6M developers choosing it directly this week, but I could not independently confirm the reason and am reporting the number as-measured rather than reweighting it. Last published 2022-06-19 and last pushed 2023-10-17 — i.e. no code changes in nearly 3 years, though the repo's `updated_at` timestamp (2026-06-20) shows some non-code GitHub activity (issues/settings) more recently. Its lower-level generator/promise-based API also requires materially more integration code than `colorthief`'s single synchronous call for a need this narrow (k≤8, no dithering, no alpha).
- **`rgbquant`** ([leeoniya/RgbQuant.js](https://github.com/leeoniya/RgbQuant.js)) is explicit in its own README that it is *not* a port of a textbook algorithm: "RgbQuant.js is not a port or implementation of any specific quantization algorithm, though some overlap is inevitable." Its own "Why?" section evaluates the alternatives it considered and rejected: Median-Cut is called "not particularly fast," Leptonica's modified median-cut is called "slow and contains major unfixed issues" (an assessment of an older codebase, not obviously still true of today's actively-maintained `quantize`/`colorthief` packages), and NeuQuant is criticized because it "can introduce edge artifacts when used with graphics since it's not designed for working with a small number of colors" — a direct, first-party warning against NeuQuant specifically for the small-palette case this project needs. The README contains **no numerical benchmarks** (confirmed by direct fetch) — its performance claims are all qualitative. 473 GitHub stars, last pushed 2024-01-24, no TypeScript types.
- **`node-vibrant`** ([Vibrant-Colors/node-vibrant](https://github.com/Vibrant-Colors/node-vibrant)) is architecturally a poor fit here: its own README describes a 3-stage pipeline — quantize, then *generate* named swatches (Vibrant, Dark Vibrant, Light Vibrant, Muted, Dark Muted, Light Muted), then filter by perceptual quality thresholds — aimed at producing a fixed set of 6 named "mood" swatches for UI theming, not an arbitrary user-chosen palette of 1-8 colors that needs to cover an entire image for a pattern. Mapping a 1-8 slider onto "pick some subset of 6 fixed named swatches" would mean fighting the library's swatch-classification step rather than benefiting from it. Actively maintained (pushed 2026-01-27), so ruled out on API fit, not on maintenance grounds.

### Hand-rolled implementation estimate
Given `quantize`'s source (~550 lines for a full MMCQ implementation with priority-queue-based box selection and a small-color-count fallback path) and that a from-scratch implementation restricted to a fixed max of 8 output colors doesn't need that general-purpose machinery, a **from-scratch median-cut implementation good enough for k=1-8** is realistically **100-200 lines of TypeScript**: build a 3D histogram or work directly on the pixel array, recursively split the box with the largest range along its longest axis up to `ceil(log2(k))` times, average each leaf box. This is a genuinely small, auditable amount of code — small enough that "hand-roll it" is a legitimate alternative to a dependency, not just theoretically possible — but it means re-solving edge cases (near-monochrome images, alpha handling, split ties) that `colorthief` has already shipped and iterated on for a savings of roughly 10 KB gzipped and one dependency.

## 3. Performance for a live preview

**No hard, dimension-specific benchmarks were found** in any of the five libraries' own docs/source for "N-megapixel photo -> k≤8 colors" timing, and the one historical benchmark I did find (Dr. Dobb's 1994 median-cut article: "about eight seconds to quantize [a] 640x480 true-color" image on a 20 MHz AT clone, [source](https://jacobfilipp.com/DrDobbs/articles/DDJ/1994/9409/9409e/9409e.htm)) is on hardware roughly three orders of magnitude slower per clock than a modern browser JS engine and is not usable as a real number for this project — I'm flagging it only to show that even decades ago the dominant cost was proportional to pixel count, matching the complexity analysis in §1, not as a timing estimate you should extrapolate from. The rest of this section is therefore **explicit reasoning from algorithmic complexity + browser API norms and documented library behavior**, not measured numbers, as instructed.

Reasoning chain:

1. **Raw pixel volume at full resolution is large.** A 4000x3000 photo is 12,000,000 pixels. `getImageData()` on a canvas that size returns a `Uint8ClampedArray` of 4 bytes/pixel = 48 MB, per the [MDN `getImageData()` reference](https://developer.mozilla.org/en-US/docs/Web/API/CanvasRenderingContext2D/getImageData). MDN's own `getContext()` docs document a dedicated `willReadFrequently` context-creation option whose entire purpose is to optimize for "a lot of read-back operations": "This will force the use of a software (instead of hardware accelerated) 2D canvas and can save memory when calling `getImageData()` frequently" ([MDN `HTMLCanvasElement.getContext()`](https://developer.mozilla.org/en-US/docs/Web/API/HTMLCanvasElement/getContext)). The existence of a browser-standard flag specifically for "you're going to call `getImageData` a lot" is itself evidence that repeated full-size readback is a known, real cost the platform expects apps to manage — it would not need to force a software rendering path if hardware-accelerated readback were already free.
2. **Any quantization pass over the full buffer is at best O(N).** Even the cheapest family (popularity histogram, §1) requires one full O(N) pass over 12M pixels per invocation. Median-cut adds an O(N log N) sort; Wu adds an O(N) moment-accumulation pass; k-means adds O(N·k) *per iteration*, typically many iterations to converge. None of these are sub-linear in N — there is no algorithm surveyed in §1 that avoids touching every sampled pixel at least once.
3. **JS engines do tens to a few hundred million simple numeric ops/sec inside a hot loop**, not billions — this is stated as a general, widely-known order-of-magnitude engineering norm for interpreted/JIT'd JS running typed-array math, not a benchmarked figure for this specific task (per the instruction to reason explicitly rather than invent precision). Under that norm, a single O(N) pass over 12M pixels alone plausibly costs on the order of several tens to low hundreds of milliseconds by itself — before adding the second/third pass that median-cut, Wu, or k-means all require, and before adding the cost of `getImageData` itself and the cost of re-rendering the resulting quantized grid to SVG (this app's existing render strategy is one SVG element per grid cell).
4. **A ~100ms "feels instant" budget for a slider-drag interaction leaves very little room for even one full-resolution pass**, let alone the multiple passes most of these algorithms need, especially since the browser main thread also has to run the slider's own re-render and the app's per-cell SVG re-render on every change.
5. **The libraries' own behavior corroborates downscaling/subsampling as the answer, not brute-forcing full resolution.** `colorthief`'s own README defaults its `quality` (pixel-sampling) option to `10`, documented as "Sampling rate (1 = every pixel, 10 = every 10th)" — i.e. **the most widely-adopted, still-actively-maintained library in this space processes roughly 1 in 10 pixels by default**, precisely to keep interactive use fast. Separately, MDN's own Canvas optimization guide explicitly tells developers to pre-scale and cache images rather than rescale on every draw: "Cache various sizes of your images on an offscreen canvas when loading as opposed to constantly scaling them in `drawImage()`" ([MDN Canvas optimization tutorial](https://developer.mozilla.org/en-US/docs/Web/API/Canvas_API/Tutorial/Optimizing_canvas)) — directly supporting a "downscale once, reuse many times" pattern rather than rescaling per interaction.
6. **Downscaling and pixel-sampling (like `colorthief`'s `quality` knob) are complementary levers, not substitutes, and downscaling is the one that must happen regardless of which library/algorithm is chosen.** `drawImage()`/canvas-based downscaling is a browser-native, hardware-accelerated primitive, cheap relative to a JS-level histogram/sort loop touching the same pixel count. Downscaling once, then optionally also subsampling within the smaller buffer, compounds the savings; but downscaling alone already gets the per-tick JS work down to a pixel count where even the least favorable algorithm (k-means, §1) becomes cheap.
7. **The downscale should be sized independently of the target grid-size control**, per the question's framing: the grid-size slider determines how many cells the *final* pixelated pattern has (e.g. 40x60 cells), which only matters at the pixel-to-cell averaging/sampling step *after* a palette already exists. The analysis-resolution downscale (a few hundred px on the long edge, chosen to comfortably exceed the largest plausible grid dimension so grid resampling always has more source detail than it needs) should happen once per crop/upload, cached, and reused across every color-count *and* every grid-size slider tick. Only a new crop or a new source photo should trigger re-downscaling.

None of the above is a substitute for measuring in the actual app once built (a quick manual check: run the chosen quantizer against a real 12MP `ImageData` in a `performance.now()`-wrapped call in a dev console) — but it is enough to commit to an architecture decision now, which is what this note is for.

## Recommendation

**Algorithm: median cut, specifically the MMCQ (Modified Median Cut Quantization) variant.** Per §1, it is the standard, lowest-risk choice for k=1-8 specifically because its cost is dominated by one O(N) ingest pass with a split phase that stays cheap at k≤8 (only `ceil(log2(k))` recursion levels — 3 at k=8, 0 at k=1), unlike k-means (whose cost multiplies by iteration count) and without octree's extra implementation complexity (which only pays off at much larger palette sizes). Popularity/histogram is not recommended as the primary algorithm — it is the single riskiest option at very small k because it can fill the whole palette with near-duplicate high-frequency colors and drop visually important minority colors entirely (§1).

**Library: adopt `colorthief` (npm `colorthief`, [github.com/lokesh/color-thief](https://github.com/lokesh/color-thief)).**
- It is the most actively maintained option checked: republished 2026-07-01 (13 days before this research), 13,599 GitHub stars, only 1 open issue, versus `image-q` (no code push since 2023-10-17), `rgbquant` (no code push since 2024-01-24), and `quantize` (no code push since 2024-10-04 — not abandoned, but visibly less active).
- Zero *required* runtime dependencies (the `sharp` peer dependency is optional and Node-only, never invoked in-browser), native TypeScript types, and a browser-native API accepting `ImageData` / `ImageBitmap` / `OffscreenCanvas` / `HTMLCanvasElement` directly — no glue code needed to pull pixels out of Canvas output.
- It implements the MMCQ algorithm family recommended above, and its `quality` (pixel-sampling) option is direct, first-party evidence (§3) that the library's own author already treats "don't touch every pixel of a full-resolution image" as the right default — which aligns with, and gives a second, complementary lever on top of, this note's downscaling recommendation below.
- Its `colorCount` option only accepts **2-20** per its documented API, not 1-8 directly — for a user-requested count of exactly **1**, call `getColorSync(img)` (single dominant-color extraction) instead of `getPaletteSync`, since 1 is outside `colorCount`'s supported range; this is a one-line special case in app code, not a blocker.
- Do not use `image-q`, `rgbquant`, or `node-vibrant` as the primary dependency: `image-q` is more powerful (it includes a Wu's-algorithm implementation) but is unmaintained at the code level (~3 years since last push) and has a heavier, promise/generator-based API for a need this narrow; `rgbquant` is explicitly self-described by its own author as "not particularly fast" and not TypeScript-typed; `node-vibrant`'s fixed 6-swatch semantic API is architecturally the wrong shape for an arbitrary 1-8 palette.
- Keep `quantize` (the bare MMCQ primitive `colorthief` is itself built on the same lineage of) in mind as a fallback if `colorthief`'s bundle size, its optional `sharp` peer dependency, or its exact API ever become a blocker — its own source already contains a small-color-count fallback path, it is dependency-free, and it is only ~1.9 KB gzipped — but there's no reason to start there when `colorthief` already wraps it with a friendlier, actively-maintained, browser-native, typed API.
- If, after building against `colorthief`, its output looks visually weak on specific test photos (MMCQ's known failure mode is treating near-duplicate popular hues as distinct when true minority colors get lost), the upgrade path is `image-q`'s `WuQuant` mode, since Wu's variance-minimizing bipartition is the algorithm with the strongest quality/speed reputation for small k among those reviewed (§1) — but treat this as a "if needed later" escape hatch, not the starting point, given `image-q`'s maintenance and API-complexity downsides above.
- Hand-rolling (§2 estimates ~100-200 lines of TypeScript for a from-scratch median-cut) is not recommended as the starting point either — it would mean re-solving edge cases `colorthief` has already shipped, for a savings of roughly 10 KB gzipped and one dependency.

**Downscaling: required, and it must happen once, at crop-commit time — not on every slider interaction.** Concretely:

1. When the user finishes cropping, draw the cropped region into an **offscreen canvas capped at roughly 200-300px on its longest side** (independent of both the original photo resolution and the current target-grid-size setting — this cap only needs to comfortably exceed the largest grid dimension the app supports, per §3.7). That's at most ~90,000 sampled pixels, several orders of magnitude cheaper per quantization pass than a 12-megapixel source.
2. Call `getImageData()` (or hand the offscreen canvas/`ImageData` straight to `colorthief`) **once** against that small offscreen canvas and cache the resulting pixel buffer in state.
3. On every color-count slider tick (and every grid-size tick, since re-mapping cached pixels to grid cells is a separate, cheap step from re-quantizing), re-run `colorthief`'s `getPaletteSync()` against that cached small buffer, not against the original photo or a fresh full-resolution `getImageData()` call. This keeps the per-tick work to a single cheap pass over ~10⁴-10⁵ pixels, which — per the order-of-magnitude reasoning in §3.3 — should comfortably clear the ~100ms budget, whereas repeating the full 12-megapixel pipeline per tick would not.
4. If profiling later shows even the capped-resolution pass isn't fast enough on low-end hardware, the next lever is lowering the sample cap further (e.g. to 100-150px) or lowering `colorthief`'s own `quality` parameter above its default of 10 — not switching algorithms.

This "downscale once, quantize the cheap cached copy on every interaction" split is the one architectural decision this research is meant to unblock; the follow-up design ticket should size the offscreen-canvas cap, decide whether to also tune `colorthief`'s `quality` parameter as a second lever, and wire the slider handlers to operate only on the cached small buffer rather than the original upload.
